# 预约记录与进出场数据整合需求文档

## 一、概述

由于威尔数据不会定时更新，需要在获取预约记录时根据车辆进出场记录（report_car_in、report_car_out）和人脸识别记录（eventRcv）来动态判断和更新预约记录的状态，并记录详细的进出场时间信息。

## 二、数据库字段扩展

### 2.1 visitor_reservation_sync 表新增字段

需要在 `visitor_reservation_sync` 表中添加以下字段：

1. **人员进出状态字段**
   - 字段名：`person_visit_status` (VARCHAR)
   - 说明：人员来访状态
   - 取值：
     - "人未来访"：初始状态，人员尚未进入
     - "人已进场"：人员已进入
     - "人已离场"：人员已离开
     - "来访中"：人员在场内（有进场记录但无离场记录）

2. **人员进出时间字段**
   - 字段名：`person_visit_times` (TEXT/JSON)
   - 说明：人员进出场时间记录，JSON格式
   - 格式：
     ```json
     [
       {
         "enterTime": "2025-01-15 10:30:00",
         "leaveTime": "2025-01-15 12:00:00"
       },
       {
         "enterTime": "2025-01-15 14:00:00",
         "leaveTime": null
       }
     ]
     ```
   - 说明：每个对象代表一次完整的进出场记录，如果只有进场没有离场，leaveTime为null

3. **车辆进出状态字段**
   - 字段名：`car_visit_status` (VARCHAR)
   - 说明：车辆来访状态
   - 取值：
     - "车未来访"：初始状态，车辆尚未进入
     - "已进场"：车辆已进入
     - "已离场"：车辆已离开

4. **车辆进出时间字段**
   - 字段名：`car_visit_times` (TEXT/JSON)
   - 说明：车辆进出场时间记录，JSON格式
   - 格式：
     ```json
     [
       {
         "enterTime": "2025-01-15 10:25:00",
         "leaveTime": "2025-01-15 12:05:00"
       },
       {
         "enterTime": "2025-01-15 14:10:00",
         "leaveTime": null
       }
     ]
     ```
   - 说明：每个对象代表一次完整的进出场记录，如果只有进场没有离场，leaveTime为null

## 三、获取预约记录时的状态判断逻辑

### 3.1 初始状态设置

当从外部系统同步预约记录时：
- 如果 `apply_state_name = "待来访"`：
  - `person_visit_status` = "人未来访"
  - `person_visit_times` = null 或空数组
  - `car_visit_status` = "车未来访"
  - `car_visit_times` = null 或空数组

- 如果 `apply_state_name = "来访中"`：
  - 需要根据车辆进出场记录判断初始状态（见3.2节）

- 如果 `apply_state_name = "已签离"`：
  - 需要根据车辆进出场记录判断初始状态（见3.3节）

### 3.2 "来访中"状态的初始判断逻辑

当 `apply_state_name = "来访中"` 时，第一次存储预约记录需要判断车辆状态：

#### 3.2.1 判断流程

**重要约束：所有查询到的进出场记录的时间必须比外部系统的预约记录的创建时间（external_create_time）要晚。**

1. **查询 report_car_out 表**
   - 根据车牌号（car_number）查询离场记录
   - 筛选条件：离场记录的 `leaveTime` > 预约记录的 `create_time`
   - 按离场时间（leaveTime）降序排序
   - 获取最新的一条离场记录

2. **判断车辆状态**

   **情况1：存在离场记录**
   - 获取最新离场记录中的进场时间（enterTime）和离场时间（leaveTime）
   - 查询 report_car_in 表，根据车牌号获取最新的进场记录
   - 筛选条件：进场记录的 `enterTime` > 预约记录的 `create_time`
   - 比较最新离场记录中的进场时间与最新进场记录中的进场时间：
     - **如果时间近似（相差不到20秒）**：
       - 说明此车辆已经离场了
       - 将离场的进出时间存储到 `car_visit_times` JSON数组中：
         ```json
         {
           "enterTime": "离场记录中的enterTime",
           "leaveTime": "离场记录中的leaveTime"
         }
         ```
       - `car_visit_status` = "已离场"
     
     - **如果最新离场记录中的进场时间 < 最新进场记录中的进场时间**：
       - 说明这个进场记录的进场时间是最新的
       - 将状态标记为"已进场"
       - 记录进场时间到 `car_visit_times` JSON数组中：
         ```json
         {
           "enterTime": "最新进场记录的enterTime",
           "leaveTime": null
         }
         ```
       - `car_visit_status` = "已进场"
     
     - **注意：不会出现离场记录中的进场时间比进场记录的进场时间大的情况**

   **情况2：不存在离场记录**
   - 查询 report_car_in 表，根据车牌号获取最新的进场记录
   - 筛选条件：进场记录的 `enterTime` > 预约记录的 `create_time`
   - 如果存在进场记录：
     - 将状态标记为"已进场"
     - 记录进场时间到 `car_visit_times` JSON数组中：
       ```json
       {
         "enterTime": "最新进场记录的enterTime",
         "leaveTime": null
       }
       ```
     - `car_visit_status` = "已进场"
   - 如果不存在进场记录：
     - `car_visit_status` = "车未来访"
     - `car_visit_times` = null 或空数组

### 3.3 "已签离"状态的初始判断逻辑

当 `apply_state_name = "已签离"` 时，第一次存储预约记录需要判断车辆状态：

**判断逻辑与3.2节"来访中"状态的判断方式完全一致。**

1. **查询 report_car_out 表**
   - 根据车牌号（car_number）查询离场记录
   - 筛选条件：离场记录的 `leaveTime` > 预约记录的 `create_time`
   - 按离场时间（leaveTime）降序排序
   - 获取最新的一条离场记录

2. **判断车辆状态**

   **情况1：存在离场记录**
   - 获取最新离场记录中的进场时间（enterTime）和离场时间（leaveTime）
   - 查询 report_car_in 表，根据车牌号获取最新的进场记录
   - 筛选条件：进场记录的 `enterTime` > 预约记录的 `create_time`
   - 比较最新离场记录中的进场时间与最新进场记录中的进场时间：
     - **如果时间近似（相差不到20秒）**：
       - 说明此车辆已经离场了
       - 将离场的进出时间存储到 `car_visit_times` JSON数组中
       - `car_visit_status` = "已离场"
     
     - **如果最新离场记录中的进场时间 < 最新进场记录中的进场时间**：
       - 说明这个进场记录的进场时间是最新的
       - 将状态标记为"已进场"
       - 记录进场时间到 `car_visit_times` JSON数组中
       - `car_visit_status` = "已进场"

   **情况2：不存在离场记录**
   - 查询 report_car_in 表，根据车牌号获取最新的进场记录
   - 筛选条件：进场记录的 `enterTime` > 预约记录的 `create_time`
   - 如果存在进场记录：
     - 将状态标记为"已进场"
     - 记录进场时间到 `car_visit_times` JSON数组中
     - `car_visit_status` = "已进场"
   - 如果不存在进场记录：
     - `car_visit_status` = "车未来访"
     - `car_visit_times` = null 或空数组

## 四、车辆进出场记录更新逻辑（多次进出处理）

对于多次进出的情况，在接收到车辆进出场记录时，需要依据车牌号码查询预约记录，根据进出场时间筛选匹配的预约记录，并更新状态和时间。

### 4.1 reportCarIn（车辆进场）处理

当接收到车辆进场记录（reportCarIn）时：

1. **查询预约记录**
   - 根据车牌号（carLicenseNumber）查询 `visitor_reservation_sync` 表
   - 根据进场时间（enterTime）筛选匹配的预约记录
   - 筛选条件：进场时间在 `gateway_transit_begin_time` 和 `gateway_transit_end_time` 范围内

2. **更新预约记录**
   - 如果找到匹配的预约记录：
     - 更新 `car_visit_status` = "已进场"
     - 在 `car_visit_times` JSON数组中添加新的进场记录：
       ```json
       {
         "enterTime": "进场时间",
         "leaveTime": null
       }
       ```
     - 判断逻辑与3.2、3.3节一致：
       - 如果数组中最后一个记录的 `leaveTime` 为 null，需要判断：
         - 如果新进场时间与上一个进场时间相近（时间差小于阈值，如1分钟），可能是重复记录，不添加
         - 否则，添加新的进场记录

### 4.2 reportCarOut（车辆离场）处理

当接收到车辆离场记录（reportCarOut）时：

1. **查询预约记录**
   - 根据车牌号（carLicenseNumber）查询 `visitor_reservation_sync` 表
   - 根据离场时间（leaveTime）筛选匹配的预约记录
   - 筛选条件：离场时间在 `gateway_transit_begin_time` 和 `gateway_transit_end_time` 范围内

2. **更新预约记录**
   - 如果找到匹配的预约记录：
     - 更新 `car_visit_status` = "已离场"
     - 判断逻辑与3.2、3.3节一致：
       - 从离场记录中获取进场时间（reportCarOut中的enterTime字段）
       - 在 `car_visit_times` JSON数组中查找对应的进场记录：
         - 查找 `leaveTime` 为 null 且 `enterTime` 与离场记录中的 `enterTime` 近似（相差不到20秒）的记录
         - 如果找到，更新该记录的 `leaveTime` 为离场记录中的 `leaveTime`
         - 如果未找到近似记录，添加新的完整记录（包含 enterTime 和 leaveTime）

3. **进场时间校验**
   - 在找到匹配的进场记录后，需要比较预约记录中的 `enterTime` 与离场记录中的 `enterTime`：
     - **如果时间相差不远（在阈值内，如30分钟内）**：
       - 不修改对应的进场时间，保持原值
     - **如果时间相差太远（超过阈值，如30分钟）**：
       - 需要修改对应的进场时间，使其与离场记录中的 `enterTime` 一致
       - 记录警告日志，标记为异常情况

## 五、人员进出场记录更新逻辑（eventRcv）

### 5.1 有车牌号的预约记录

#### 5.1.1 人员进出处理

当接收到人脸识别进出记录（eventRcv，eventType = 196893 或 197162 或 198914）时：

1. **查询人员信息**
   - 根据工号/学号（personId）查询人员姓名
   - 根据姓名查询对应的预约记录
   - 根据事件发生时间（eventTime）筛选匹配的预约记录
   - 筛选条件：事件时间在 `gateway_transit_begin_time` 和 `gateway_transit_end_time` 范围内

2. **更新人员进出状态**
   - 如果找到匹配的预约记录：
     - 根据方向（direction）判断是"进"还是"出"
     - 如果是"进"：
       - 更新 `person_visit_status` = "人已进场"
       - 在 `person_visit_times` JSON数组中添加新的进场记录：
         ```json
         {
           "enterTime": "事件发生时间",
           "leaveTime": null
         }
         ```
     - 如果是"出"：
       - 更新 `person_visit_status` = "人已离场"
       - 在 `person_visit_times` JSON数组中查找对应的进场记录：
         - 查找 `leaveTime` 为 null 且 `enterTime` 最接近离场时间的记录
         - 如果找到，更新该记录的 `leaveTime`
         - 如果未找到，添加新的完整记录

#### 5.1.2 车辆进出处理

- 车辆进出处理逻辑同第四章（reportCarIn/reportCarOut）

### 5.2 无车牌号的预约记录

对于没有车牌号的预约记录，只能通过刷人脸进入/出去：

1. **处理逻辑**
   - 当接收到人脸识别进出记录时：
     - 根据工号/学号查询人员姓名
     - 根据姓名查询预约记录（无车牌号）
     - 根据事件发生时间筛选匹配的预约记录

2. **状态更新**
   - 第一次进或出的时候：
     - 将状态标记为"人已来访"（`person_visit_status` = "人已进场" 或 "人已离场"）
     - 来访时间写成JSON格式：
       ```json
       [
         {
           "enterTime": "第一次进场时间",
           "leaveTime": "第一次离场时间（如果有）"
         },
         {
           "enterTime": "第二次进场时间",
           "leaveTime": null
         }
       ]
       ```
   - 一个数组代表一次进出，每个数组中的开始和结束时间对应人员进出的发生时间

3. **多次进出处理**
   - 可能出现一条预约记录多次进出的情况
   - 每次进出都在JSON数组中添加或更新对应的记录
   - 通过当前时间和最新的进出场时间判断当前状态

## 六、时间格式说明

### 6.1 JSON时间格式

所有时间字段统一使用格式：`"yyyy-MM-dd HH:mm:ss"`

示例：`"2025-01-15 10:30:00"`

### 6.2 时间比较规则

- 使用标准的时间比较方法
- 时间差计算使用毫秒或秒为单位
- **时间近似判断阈值：20秒**（用于判断离场记录中的进场时间与进场记录中的进场时间是否匹配）
- 重复记录判断阈值：1S（1秒）（用于判断是否为重复的进出场记录）
- 时间相差太远判断阈值：30分钟（1800秒）（用于异常情况判断）

## 七、异常情况处理

### 7.1 数据不一致

- 如果离场记录中的 `enterTime` 与预约记录中的 `enterTime` 相差太远：
  - 记录警告日志
  - 可以选择更新预约记录中的进场时间，或保持原值并标记异常

### 7.2 重复记录

- 如果检测到重复的进出场记录（时间相近）：
  - 不重复添加
  - 记录日志

### 7.3 找不到匹配的预约记录

- 如果根据车牌号或姓名找不到匹配的预约记录：
  - 记录日志
  - 不进行更新操作

## 八、实现要点

### 8.1 数据查询优化

- 使用索引优化车牌号和姓名的查询
- 使用时间范围查询优化匹配逻辑

### 8.2 JSON数据处理

- 使用JSON库（如FastJSON）进行JSON数组的解析和更新
- 确保JSON格式的正确性和一致性

### 8.3 并发处理

- 考虑并发更新的情况
- 使用数据库锁或乐观锁机制避免数据冲突

### 8.4 日志记录

- 记录所有关键操作和异常情况
- 便于问题排查和数据追溯

## 九、数据流程图

### 9.1 车辆进出场更新流程

```
reportCarIn/reportCarOut
    ↓
根据车牌号查询预约记录
    ↓
根据时间范围筛选匹配记录
    ↓
更新car_visit_status和car_visit_times
    ↓
保存到数据库
```

### 9.2 人员进出场更新流程

```
eventRcv（人脸识别）
    ↓
根据工号/学号查询人员姓名
    ↓
根据姓名查询预约记录
    ↓
根据时间范围筛选匹配记录
    ↓
更新person_visit_status和person_visit_times
    ↓
保存到数据库
```

### 10.1 基本场景

1. 待来访状态预约记录的初始设置

3. 已签离状态预约记录的初始状态判断

## 十一、注意事项

1. **数据一致性**：确保进出场记录的配对正确
2. **时间精度**：统一使用相同的时间格式和精度
3. **性能优化**：大量数据查询时注意性能优化
4. **错误处理**：完善的异常处理和日志记录
5. **向后兼容**：新增字段需要考虑旧数据的兼容性
